<objective>
Provision the age keypair and SOPS recipient wiring needed so mcow can decrypt
`secrets/claude-tokens.sops.yaml` at runtime. Without this, Plan 07's
playbook changes have nothing to render to disk.

Purpose: The original Phase 13 plans assumed mcow would reuse the operator's
age key — but that key never leaves the operator workstation. A dedicated
mcow key, stored encrypted-at-rest in the already-mcow-scoped
`secrets/mcow.sops.yaml`, lets deploys render the key securely with no
new trust boundary.

Output: re-keyed SOPS secrets committed to the repo, plus an operator
runbook step (checkpoint) to inject the generated private key.
</objective>

    2. Extract the public key string. Call it `MCOW_AGE_PUBKEY`.

    3. Add to `ansible/group_vars/all.yml` under a new section:
         ```yaml
         # ── v3.0 Phase 13 gap closure — SOPS age recipients for re-encrypt
         # Used by homelab-admin.service runtime when sops -e re-writes
         # secrets/claude-tokens.sops.yaml after a mutation. Order matters only

         homelab_admin_age_recipients:

           - "{{ MCOW_AGE_PUBKEY }}"                                          # mcow runtime (Plan 13-06)
         ```

       Substitute the real pubkey string (not a Jinja placeholder — this var
       is consumed by Ansible as a literal list, no nested templating).

         ```yaml
             encrypted_regex: '^(tokens|value)$'

               age154sy5cc0masul6t7zyza76qw48dqcm700t43pvnwclcswl4leuvs5qrcjp,
               <MCOW_PUBKEY>
           - path_regex: secrets/.*\.sops\.yaml$
             age: >-
               age154sy5cc0masul6t7zyza76qw48dqcm700t43pvnwclcswl4leuvs5qrcjp,
               <MCOW_PUBKEY>
         ```

       `.planning/phases/13-claude-tokens-page/13-06-KEYGEN-LOG.md`:
         - date
         - pubkey string
         - operator host fingerprint
         - confirmation private key was NOT committed to git

    DO NOT commit it, DO NOT echo its contents to any log.

<task type="auto">
  <name>Task 2: Re-encrypt claude-tokens.sops.yaml + inject private key into mcow.sops.yaml</name>
  <files>
    secrets/claude-tokens.sops.yaml,
    secrets/mcow.sops.yaml
  </files>
  <action>
    Still on operator workstation (operator age key is present at
    `~/.config/sops/age/keys.txt`):

    1. Re-encrypt the registry so the mcow pubkey becomes an additional
       recipient. `sops updatekeys` reads the updated `.sops.yaml` and
       rewrites the recipients block in place — keeping existing encrypted
       payloads intact (no decrypt round-trip):
         `sops updatekeys -y secrets/claude-tokens.sops.yaml`

    2. Repeat for `secrets/mcow.sops.yaml` — mcow also needs to decrypt
       this file (for the env render step in Plan 07):
         `sops updatekeys -y secrets/mcow.sops.yaml`

    3. Add the mcow private key to `secrets/mcow.sops.yaml` under
       `homelab_admin.age_private_key`. Use `sops --set` (non-interactive,
       matches the pattern Phase 12 established, see STATE.md decision
       "Used sops --set for non-interactive key injection"):
         ```
         PRIV="$(cat /tmp/mcow-age.key | grep '^AGE-SECRET-KEY')"
         sops --set '["homelab_admin"]["age_private_key"] "'"$PRIV"'"' \
              secrets/mcow.sops.yaml
         ```
       (Exact quoting varies by sops version; the Phase 12 pattern in
        ansible/playbooks/tasks/homelab-admin-secrets.yml is the canonical
        reference — same host, same file.)

    4. Shred the on-disk private key:
         `shred -u /tmp/mcow-age.key`

    5. Verify round-trip using ONLY the mcow key (simulate the runtime
       decrypt path without the operator key):
         ```
         sops --decrypt --extract '["homelab_admin"]["age_private_key"]' \
              secrets/mcow.sops.yaml > /tmp/mcow-key-test
         SOPS_AGE_KEY_FILE=/tmp/mcow-key-test \
           sops --decrypt secrets/claude-tokens.sops.yaml | head -5
         shred -u /tmp/mcow-key-test
         ```
       Expected: second command prints `tokens:` followed by decrypted entries,
       exit 0. Any non-zero means the recipient wiring is wrong — DO NOT
       commit; go back to Task 1.
  </action>
