# 17.1-02 iGPU Probe — Tower HW Transcode Gate

**Date:** 2026-04-18
**Host:** tower (Proxmox, i7-8700, kernel 6.17.2-1-pve)
**Purpose:** Determine why `/dev/dri/renderD128` is NVIDIA (nouveau) instead of Intel UHD 630, blocking D-17.

## Commands Run

```bash
ssh root@tower
dmesg | grep -iE 'i915|intel.*(gpu|display)' | head -50
lspci -nn | grep -iE 'vga|display|3d|nvidia|intel'
ls -la /sys/class/drm/
for v in /sys/class/drm/card*/device/vendor; do echo "$v: $(cat $v)"; done
cat /sys/class/drm/card*/device/device 2>/dev/null
lsmod | grep i915
modprobe -n -v i915
grep -m1 'model name' /proc/cpuinfo
```

## Raw Outputs (key excerpts)

### lspci — no Intel VGA/Display device present
```

00:01.0 PCI bridge [8086:1901]      — PCIe x16 root port
00:1f.x (various Intel PCH devices — audio, SMBus, SPI, LAN)

```
**No `00:02.0 VGA [8086:3e9b]` (UHD 630) entry.** The CPU's integrated GPU is not exposed to the OS.

### /sys/class/drm — only NVIDIA card0
```
card0 -> devices/pci0000:00/0000:00:01.0/0000:01:00.0/drm/card0
renderD128 -> devices/pci0000:00/0000:00:01.0/0000:01:00.0/drm/renderD128
card0/device/vendor: 0x10de   (NVIDIA)
card0/device/device: 0x1f08   (TU106 RTX 2060)
```
`renderD128` resolves to the NVIDIA GPU, driven by `nouveau` (seen in dmesg module list).

### i915 module — available, never bound
```
lsmod | grep i915   → (empty, i915 not loaded)
modprobe -n -v i915 → insmod drm_buddy.ko; insmod i915.ko   (module exists)
```
Module present in kernel tree; not auto-loaded because no matching PCI device (`8086:3e9b`) is on the bus.

### CPU
```
Intel(R) Core(TM) i7-8700 CPU @ 3.20GHz   — has UHD 630 iGPU (device id 3e92/3e9b)
```

## Diagnosis

**BIOS-disabled iGPU.** Verdict: not dead, not a kernel/driver issue.

Evidence chain:
1. CPU is i7-8700 → physically has UHD 630 silicon.
2. No Intel `[8086:xxxx]` VGA/Display PCI function exists in `lspci` (would appear at `00:02.0` if enabled).
3. `i915` kernel module is present and loadable but has no device to bind.
4. Only GPU enumerated via DRM is the NVIDIA dGPU at `01:00.0` (nouveau-driven).
5. Motherboard (B360 chipset, `[8086:a308]`) supports iGPU+dGPU coexistence; the iGPU is simply switched off at firmware level, most likely because the BIOS defaults to "Primary Display = PEG" and disables IGD when a dGPU is detected.

This is the typical behavior of consumer motherboards with a discrete card installed: iGPU is auto-disabled unless explicitly enabled via "IGD Multi-Monitor" / "Integrated Graphics" setting.

## BIOS Action — Setting to Toggle

Tower is likely on an ASUS/MSI/Gigabyte B360 board. Enter BIOS (usually `Del` or `F2` at POST) and look under **Advanced / System Agent (SA) Configuration / Graphics Configuration** (naming varies by vendor) for one or more of:

| Vendor        | Setting name                                                   | Value            |
| ------------- | -------------------------------------------------------------- | ---------------- |
| ASUS          | `Primary Display` / `iGPU Multi-Monitor`                       | Auto→PCIe; **iGPU Multi-Monitor: Enabled** |
| MSI           | `Integrated Graphics Configuration` → `Initiate Graphic Adapter` + `IGD Multi-Monitor` | Keep PEG as primary; **IGD Multi-Monitor: Enabled** |
| Gigabyte      | `Internal Graphics`                                            | **Enabled** (not Auto) |
| ASRock        | `Primary Graphics Adapter` + `IGPU Multi-Monitor`              | PCI Express; **IGPU Multi-Monitor: Enabled** |

Leave the NVIDIA dGPU as primary display output (keeps Proxmox console on whichever head is wired). The key is: force the iGPU to remain **enabled and enumerated on the PCI bus** even when a dGPU is present.

Optional but recommended: also enable **VT-d / IOMMU** if not already on (future passthrough work).

## Post-Reboot Resume Checklist

After the BIOS change and reboot, run these on tower to confirm iGPU is now present before resuming plan 17.1-02:

```bash
# 1. Intel VGA must now appear (expect 00:02.0)
ssh root@tower "lspci -nn | grep -iE '00:02|intel.*(vga|display)'"
# Expected: 00:02.0 VGA compatible controller [0300]: Intel Corporation CoffeeLake-S GT2 [UHD Graphics 630] [8086:3e9b]

# 2. i915 module loaded automatically
ssh root@tower "lsmod | grep i915 && dmesg | grep -i i915 | tail -20"
# Expected: i915 present; dmesg shows "i915 0000:00:02.0: [drm] ... GT0" etc.

# 3. Two DRM cards now (card0=nvidia, card1=intel) with two renderD*
ssh root@tower "ls -la /sys/class/drm/ | grep -E 'card|render'"
# Expected: card0 + card1, renderD128 + renderD129
# renderD128 historically binds to whichever GPU enumerates first — may SWAP; verify by vendor:
ssh root@tower "for v in /sys/class/drm/card*/device/vendor; do echo \"\$v: \$(cat \$v)\"; done"
# Look for 0x8086 on one of them

# 4. Identify which render node is Intel (plan's LXC passthrough targets Intel)
ssh root@tower "readlink -f /sys/class/drm/renderD128; readlink -f /sys/class/drm/renderD129 2>/dev/null"
# Match against lspci BDF 0000:00:02.0 to pick the Intel node

# 5. On CT 101: after tower reboot, restart the container and re-verify
ssh root@tower "pct reboot 101 && sleep 10 && pct exec 101 -- vainfo"
# Expected: "iHD driver" or "i965 driver", profiles listed, no "error: can't connect"

# 6. Resume plan 17.1-02: run verify-jellyfin-lxc.sh and confirm D-17 (HW transcode) passes
```

**If renderD nodes swap** (Intel gets 128, NVIDIA gets 129), the LXC config `dev0: /dev/dri/renderD128` may need to be updated to the Intel node — plan's passthrough must target device id `8086:3e9b`, not `10de:1f08`.

**If Intel VGA still absent after BIOS toggle:** capture fresh `lspci -vv` and BIOS screenshots; silicon may be fused off (rare for retail i7-8700) or motherboard lacks iGPU routing.
