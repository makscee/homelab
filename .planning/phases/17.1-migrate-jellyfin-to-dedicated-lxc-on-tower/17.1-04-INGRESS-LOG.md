## Verification: FAILED — architectural blocker discovered

**Symptom:** After DNAT flip, curl to `tower:8096` times out. CT 101 jellyfin is running
(`ss -tlnp` on CT 101 shows listener on `0.0.0.0:8096`), ARP resolves, packets arrive at
CT 101's `eth0` (RX counters increment during ping). No iptables/nftables rule drops them.

**Root cause:** Tailscale on CT 101 installs policy-routing rule `5270 from all lookup 52`,
and table 52 contains `10.10.20.0/24 dev tailscale0` (accepted subnet route advertised by
another tailnet peer). Reply packets to `10.10.20.1` are routed out `tailscale0` instead of
`eth0`:

```
# pct exec 101 -- ip route get from 10.10.20.11 to 10.10.20.1
10.10.20.1 from 10.10.20.11 dev tailscale0 table 52 uid 0
```

Tower (on `10.10.20.1`, not a tailnet route advertiser for 10.10.20/24 itself) never sees
the reply. Result: TCP SYN → no SYN-ACK → timeout.

**This is a cross-LXC configuration issue NOT addressed by Plan 02 (CT 101 provisioning).**
The Plan-02 tailscale bring-up on CT 101 used `--accept-routes`, which absorbed the
`10.10.20.0/24` subnet-route advertisement from another tailnet peer and now shadows the
L2-local route via eth0.

### Fix options (operator decision required — Rule 4 escalation)

| Option | Action | Trade-off |
|--------|--------|-----------|
| A | `tailscale up --accept-routes=false` on CT 101 | Simple. CT 101 loses access to subnet-routed nets from other peers (probably not needed on a jellyfin box). |
| B | Add ip rule on CT 101: `ip rule add to 10.10.20.0/24 lookup main pref 100` | Surgical; persists via systemd-networkd or `/etc/network/if-up.d/` hook. |
| C | Remove `10.10.20.0/24` advertisement from whichever peer advertises it | Affects whole tailnet; unclear which peer advertises. |

**Recommendation: Option A.** Jellyfin CT 101 is a leaf node; no reason to accept routes.

### Rollback currently applied?

No — the DNAT flip itself is correct and persisted. Rolling back would restore traffic to
`10.10.20.10` (docker-tower, STOPPED per cutover commit 0ecdbf5) — that would NOT help.
External users are offline regardless until Option A/B is applied on CT 101.

Temporary tweak made during diagnosis: set `rp_filter=0` on CT 101 via sysctl (not
persisted). Does NOT fix the issue — routing-table shadowing is the real cause.
