# Phase 17.1: Migrate Jellyfin to dedicated LXC on tower — Context

**Gathered:** 2026-04-18
**Status:** Ready for planning

<domain>
## Phase Boundary

Move Jellyfin off docker-tower LXC 100 (Docker) to a new dedicated Proxmox LXC on `tower` running Jellyfin natively (deb package). New LXC owns `/dev/dri` (Intel QSV) exclusively and mounts the media libraries read-only. Migration preserves users, library, metadata, and watch progress. Old Docker Jellyfin stays as hot standby for 1 week, then is removed from the media compose.

**Explicitly out of scope:**
- Changes to *arr suite, qBittorrent, Navidrome, Filebrowser — they keep running on docker-tower LXC 100
- Moving the WDC/SEA disks (they stay bindmounted to LXC 100 for *arr + downloads)
- VoidNet user migration/re-onboarding beyond verifying auth still works

</domain>

<decisions>
## Implementation Decisions

### Motivation (drives scope + verify)
- **D-01:** Primary goals: (a) cleaner iGPU ownership by giving `/dev/dri` to a single LXC, (b) resource isolation from the busy docker-tower stack, (c) hope to reduce playback buffering. Buffering improvement is the user-visible success signal.

### LXC provisioning
- **D-02:** New LXC on tower — **CT 101**, hostname `jellyfin`, Tailscale-joined. Resources: **4 cores / 8 GB RAM / 32 GB rootfs**. OS: Debian 12 (match existing LXCs). Unprivileged.
- **D-03:** Network: single NIC on `vmbr1` at `10.10.20.11/24` (gw 10.10.20.1). No vmbr0 NIC needed — tower handles external ingress (see D-11). Tailscale on top of vmbr1.
- **D-04:** Provisioning pattern: **manual `pct create` on tower + commit `servers/tower/lxc-101-jellyfin.conf` to repo + Ansible role for post-create config**. Mirrors existing docker-tower / cc-* pattern. No community.proxmox collection on the create step.

### Runtime + install
- **D-05:** Jellyfin installed from **official Jellyfin deb repo** (`repo.jellyfin.org`), systemd unit, running as `jellyfin:jellyfin`. No Docker inside the new LXC.
- **D-06:** Post-install: Ansible playbook manages the deb repo + pin, installs `jellyfin` + `jellyfin-ffmpeg*`, renders `/etc/jellyfin/*` config, owns `/var/lib/jellyfin` perms, writes the tmpfs unit (D-09), and handles service restarts.

### Storage + transcode
- **D-07:** Media libraries visible via **Proxmox bindmounts mp0/mp1 read-only** — same pattern as LXC 100:
  - `mp0: /mnt/pve/wdc,mp=/media/wdc,ro=1`
  - `mp1: /mnt/pve/sea,mp=/media/sea,ro=1`
- **D-08:** Jellyfin UID/GID inside new LXC must resolve the same uid ownership as LXC 100 sees today so metadata scans don't choke on permissions. Research (RESEARCH.md) must confirm the uid-mapping for unprivileged LXC + bindmount read path; fall back to `lxc.idmap` tweaks if needed.
- **D-09:** Transcode scratch: **tmpfs mounted at `/var/cache/jellyfin/transcodes`**, ~4 GB, via `systemd.mount` unit. Jellyfin config points transcode temp path there. Matches current `/mnt/transcode` pattern on docker-tower.

### iGPU / hardware transcoding
- **D-10:** **Unprivileged LXC + `/dev/dri` bind + render-group gid mapping** (same approach as LXC 100 conf: `lxc.cgroup2.devices.allow: c 226:* rwm` + `lxc.mount.entry: /dev/dri dev/dri none bind,optional,create=dir`). Jellyfin configured for VA-API/QSV via `jellyfin-ffmpeg`. Privileged fallback is rejected unless research shows unprivileged render-group cannot work.

### External ingress
- **D-11:** `jellyfin.makscee.ru` resolves to the home router's public IP. Router already forwards **port 22098** to `tower` host. Tower must proxy `:22098` → `lxc-101:8096` (over vmbr1 `10.10.20.11`). Research must confirm what's on tower:22098 today (nginx/Caddy/raw DNAT) and pick the minimum change. TLS termination location is a research question — do not assume.
- **D-12:** Internal access: Jellyfin reachable via Tailnet hostname (`jellyfin` / Tailnet IP) for operator + admin workflows.

### State migration
- **D-13:** **rsync-based migration** of Jellyfin state from docker-tower → new LXC:
  - Stop old container cleanly (not just pause) before rsync to avoid SQLite corruption.
  - Source: `/opt/docker-configs/shared/jellyfin/` on LXC 100 (this is the container's `/config` volume — contains `config/`, `data/`, `metadata/`, `cache/`, `log/`, `plugins/`, `root/`).
  - Target paths on new LXC: Jellyfin deb expects `/etc/jellyfin/` (config), `/var/lib/jellyfin/` (data + metadata + cache + plugins), `/var/log/jellyfin/` (logs). Research must produce the exact source→target mapping — paths differ between the Docker LinuxServer-style layout and the native deb layout.
  - chown `jellyfin:jellyfin` recursively after rsync.
  - Start service; verify DB schema version matches (no automatic upgrade surprises).

### Cutover + rollback
- **D-14:** **Hot standby for 1 week.** Old jellyfin container keeps its definition in `servers/docker-tower/media/docker-compose.yml` for the soak window. Port 8096 stays bound on LXC 100 for that week (DNS points elsewhere, so no user traffic).
- **D-15:** After 1 week (target cleanup date: **2026-04-25**) and success-criteria green (see verify), remove the `jellyfin:` service block from `media/docker-compose.yml` in a dedicated commit, keep `/opt/docker-configs/shared/jellyfin/` on LXC 100 for one additional week as an offline backup, then purge.
- **D-16:** Rollback path during soak: stop Jellyfin systemd unit on LXC 101, start old container on LXC 100, repoint tower:22098 proxy to `docker-tower:8096`. Since state drifts the moment the new LXC accepts writes, rollback loses any new-LXC-only activity — acceptable for a 1-week soak.

### Verification (success criteria for cleanup)
- **D-17:** Must all pass before purging old container:
  1. **HW transcode confirmed** — `vainfo` inside LXC 101 shows the iGPU, and a Jellyfin playback session on a client that needs transcoding shows hardware acceleration in the Jellyfin dashboard + ffmpeg logs.
  2. **Playback performance** — 1080p and 4K streams to the user's real test client play without buffering under typical network conditions. This is the primary motivation.
- **D-18:** Also nice-to-have (not blocking cleanup): users can log in with existing credentials, recent watch progress survived migration.

### Claude's Discretion
- Exact Ansible role structure for the Jellyfin install (new role vs extend existing playbook pattern).
- Systemd tmpfs unit wording + exact size (can bump 4G if research suggests otherwise).
- UID mapping specifics (`lxc.idmap` tweaks) if the default unprivileged mapping doesn't work with the RO bindmounts.
- Whether to disable Jellyfin's auto-update in favor of Ansible-managed updates (recommend yes, for IaC consistency).

</decisions>

### Reusable assets
- `servers/tower/lxc-*.conf` template pattern — direct copy/modify for CT 101.

### Established patterns
- **LXC conf committed to repo**: manual `pct create`, then the final `.conf` is copied from `/etc/pve/lxc/<id>.conf` on tower into `servers/tower/` and committed. This phase follows that.
- **Per-service config dir convention**: on docker-tower it's `/opt/docker-configs/shared/<service>` — NOT applicable on new LXC; native deb uses `/etc/jellyfin` + `/var/lib/jellyfin`. Keep both conventions documented so the service inventory stays readable.
- **Tailscale join**: LXCs auto-join via existing bootstrap role (mirror cc-worker / animaya-dev pattern).

### Integration points
- `tower`:22098 ingress path (router-forwarded) — requires tower-host-level change, not just LXC-internal. Plan must list the specific file on tower that gets edited (nginx/Caddy config or iptables rule) — this is the only cross-LXC change.
- Monitoring: Prometheus scrape target list + Grafana dashboard for new host. If there's an existing "media server" Grafana dashboard tied to docker-tower, it likely needs a second panel or a relabel for jellyfin host.

<specifics>
## Specific Ideas

- User's phrasing: *"for a chance that it will stop buffering so much"* — buffering relief is the emotional primary goal. Verify test must include real playback by the user, not just synthetic throughput.
- User pre-configured DNS + router forward: `jellyfin.makscee.ru` → public IP, router :22098 → tower. Downstream work must respect what's already live, not re-do DNS.
- Keep old Jellyfin stack dormant but restorable for **1 full week**. Cleanup is explicitly scheduled (target 2026-04-25).

</specifics>

<deferred>
## Deferred Ideas

- **Move WDC/SEA disks entirely to Jellyfin LXC** — user rejected; would break *arr/qBittorrent on LXC 100. Revisit only if media-stack re-architecture becomes a phase.
- **NFS between LXC 100 and LXC 101 for /media** — rejected; Proxmox bindmounts are simpler and faster.
- **Jellyfin backup-plugin-based migration** — rejected in favor of rsync for fidelity. Consider for future routine backups (separate phase).
- **Privileged LXC for iGPU** — only fallback if unprivileged + render-group fails during research; if used, document the security tradeoff in RESEARCH.md.
- **Jellyfin auto-update** — recommend disabling in favor of Ansible-managed upgrades; track as a planning-phase decision.
- **Old container cleanup commit (purge compose block + config dir)** — deferred todo for 2026-04-25 after soak.

</deferred>
