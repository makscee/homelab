# Phase 17.1 — Migrate Jellyfin to Dedicated LXC on Tower — Research

## Summary

Move Jellyfin off docker-tower (LXC 100, Docker) to a new dedicated unprivileged Proxmox LXC (**CT 101**, `jellyfin`, Debian 12, 4C/8G/32G, `vmbr1 10.10.20.11/24`) running Jellyfin natively from the official Jellyfin deb repo. Old container stays as 1-week hot standby (cleanup target **2026-04-25**).

**Primary recommendation:** Use the modern Proxmox 8.2 `dev0: /dev/dri/renderD128,gid=<host-render-gid>` one-liner for iGPU passthrough (replaces the legacy `lxc.cgroup2.devices.allow` + `lxc.mount.entry` pair used on LXC 100). Install Jellyfin via the official `repo.jellyfin.org` apt repo (installs `jellyfin` metapackage which pulls `jellyfin-server` + `jellyfin-web` + `jellyfin-ffmpeg7`). Migrate state by rsyncing the Docker `/opt/docker-configs/shared/jellyfin/` tree into the native deb paths using the documented sub-dir→path mapping. Ingress via tower-side nginx (needs inspection — see Open Q 1).

### Anti-Patterns to Avoid
- **Editing `/etc/pve/lxc/101.conf` by hand then forgetting to re-capture into repo.** LXC 100's current conf in the repo is out of date risk — always `ssh root@tower 'pct config 101' > servers/tower/lxc-101-jellyfin.conf` immediately after any tower-side change.
- **rsync with source container still running.** SQLite open files corrupt silently. Must `docker stop jellyfin` (not pause) on LXC 100 before rsync step. [Pitfall #1]
- **Using `jellyfin/jellyfin:latest` image convention to infer LinuxServer paths.** Current compose uses OFFICIAL image (`jellyfin/jellyfin:latest`) not LinuxServer — paths inside `/config` follow the OFFICIAL layout (see §Key Findings #4). Do not reach for LinuxServer migration guides; they're a red herring.
- **Assuming `renderD128` is always the Intel device.** tower may have renderD128=Intel or renderD129 after a kernel update. Detect at runtime (existing `scripts/enable-jellyfin-qsv.sh` does this with `/sys/class/drm/renderD*/device/vendor == 0x8086`).

| Problem | Don't build | Use instead | Why |
|---------|-------------|-------------|-----|
| Docker → native path mapping | Custom rsync wildcarding | The explicit sub-dir mapping in §Key Findings #4 | Fewer moves = fewer ways to corrupt SQLite |

## Key Findings

### 1. LXC creation command

Based on tower's README and `lxc-204-cc-worker.conf` (closest unprivileged analog):

```bash
ssh root@tower 'pct create 101 local:vztmpl/debian-12-standard_12.<latest>_amd64.tar.zst \
    --hostname jellyfin \
    --cores 4 \
    --memory 8192 \
    --swap 512 \
    --rootfs toshi:32 \
    --net0 name=eth0,bridge=vmbr1,gw=10.10.20.1,ip=10.10.20.11/24,type=veth \
    --nameserver 1.1.1.1 \
    --ostype debian \
    --unprivileged 1 \
    --features nesting=1,keyctl=1 \
    --onboot 1 \
    --timezone Europe/Moscow \
    --tags jellyfin,media,homelab'
```

**Notes:**
- **`nesting=1`**: enables systemd inside unprivileged LXC (required by Tailscale + tmpfs unit).
- **`keyctl=1`**: needed for some systemd services on unprivileged LXCs (kernel keyring). [CITED: Proxmox docs — Unprivileged_LXC_containers]
- **`fuse=1`**: NOT needed; no FUSE mounts.
- **rootfs on `toshi:32`**: 32 GB per D-02. Matches storage pool convention (LXC 100 also on `toshi`).
- **Template**: query `pveam list local | grep debian-12` on tower at plan time to pin the exact version string.
- **No vmbr0 NIC** (D-03 explicit).
- **No swap reduction** (512M matches cc-worker).
- Password/SSH key: set via `--password` + `--ssh-public-keys` (use existing operator pubkey, SOPS-encrypted), **not shown above** to avoid leaking in plan — plan task should source from SOPS.

### 2. LXC config additions for iGPU + mp0/mp1

Append to `/etc/pve/lxc/101.conf` (also `pct set 101 ...` equivalents exist):

```
mp0: /mnt/pve/wdc,mp=/media/wdc,ro=1
mp1: /mnt/pve/sea,mp=/media/sea,ro=1
dev0: /dev/dri/renderD128,gid=<HOST_RENDER_GID>
```

**Critical: Proxmox 8.2+ `dev0:` is the current idiomatic approach**, replacing the legacy pair:
```
# LEGACY (what LXC 100 has) — not recommended for new LXC
lxc.cgroup2.devices.allow: c 226:* rwm
lxc.mount.entry: /dev/dri dev/dri none bind,optional,create=dir
```

The `dev0:` form automatically creates a correctly-permissioned device node inside the unprivileged LXC, sidestepping idmap complexity for the device itself. [CITED: Proxmox forum #158325, tteck Proxmox Discussion #2850]

**`<HOST_RENDER_GID>` determination:** probe at plan time:
```bash
ssh root@tower 'getent group render | cut -d: -f3'
# Debian 12 default: 107. Can also be 104 or 993 depending on package install order.
```

**Inside the LXC**, after Jellyfin install, add `jellyfin` user to `render` and `video` groups with the SAME GIDs as on the host (unprivileged LXC's gid-map means gid 107 inside = gid 100107 on host; the `dev0:` `gid=` parameter aligns device ownership). If groups do not exist yet inside LXC with matching GIDs, create them:
```bash
groupadd -g 107 render  # if missing; match host gid
groupadd -g 44 video    # if missing; match host gid
usermod -aG render,video jellyfin
systemctl restart jellyfin
```
[CITED: jellyfin.org intel hwaccel docs, sadrithmina.de Proxmox guide]

**Card node (`/dev/dri/card0`):** Jellyfin only needs `renderD128` for compute; `card0` is for display output and should NOT be passed through. Single `dev0:` entry is sufficient.

**Fallback:** If `dev0:` produces wrong uid/gid (rare), use legacy syntax copied verbatim from LXC 100's conf — keeping this phase unblocked. This is the "privileged fallback" exit per D-10 but WITHOUT privileged mode.

### 4. Docker → deb state migration path map

**Current container is `jellyfin/jellyfin:latest`** (official image per `servers/docker-tower/media/docker-compose.yml` line 24) — **NOT LinuxServer**. This is critical. The official image puts everything under `/config` in a flat layout. [CITED: jellyfin.org issue #3717]

Docker `/config` (= host `/opt/docker-configs/shared/jellyfin/`) layout:
```
/opt/docker-configs/shared/jellyfin/
├── config/       — XML configs (system.xml, encoding.xml, network.xml, branding.xml, dlna/, users/, libraries/)
├── data/         — SQLite DBs (library.db, jellyfin.db), users, scheduled task state
├── metadata/     — image cache (posters, thumbs, artist images)
├── cache/        — non-transcode caches (sizing, subtitle, etc.)
├── log/          — log files
├── plugins/      — installed plugins + their configs
└── root/         — Jellyfin "root" virtual FS (usually small)
```

**Mapping to native deb** [CITED: jellyfin.org/docs/general/administration/migrate + jellyfin.org env vars]:

| Source (Docker) | Target (deb) | Notes |
|-----------------|--------------|-------|
| `.../config/*` | `/etc/jellyfin/` | Copy all XML files and subdirs. |
| `.../data/*` | `/var/lib/jellyfin/data/` | SQLite DBs live HERE on native deb (not at top level). |
| `.../metadata/*` | `/var/lib/jellyfin/metadata/` | Large; primary time cost of rsync. |
| `.../plugins/*` | `/var/lib/jellyfin/plugins/` | Plugin configs sit alongside; copy verbatim. |
| `.../root/*` | `/var/lib/jellyfin/root/` | Small; copy verbatim. |
| `.../cache/*` | `/var/cache/jellyfin/` | Safe to skip (Jellyfin regenerates), but copying preserves warmth. |
| `.../log/*` | `/var/log/jellyfin/` | Optional (history only). |

**rsync invocation** (run from tower or controller, source = docker-tower):
```bash
# Controller orchestrates; or ssh root@tower 'rsync ...'
rsync -aHAX --info=progress2 \
  /opt/docker-configs/shared/jellyfin/config/    root@jellyfin:/etc/jellyfin/
rsync -aHAX --info=progress2 \
  /opt/docker-configs/shared/jellyfin/data/      root@jellyfin:/var/lib/jellyfin/data/
rsync -aHAX --info=progress2 \
  /opt/docker-configs/shared/jellyfin/metadata/  root@jellyfin:/var/lib/jellyfin/metadata/
rsync -aHAX --info=progress2 \
  /opt/docker-configs/shared/jellyfin/plugins/   root@jellyfin:/var/lib/jellyfin/plugins/
rsync -aHAX --info=progress2 \
```

### 10. UID mapping for unprivileged LXC + RO bindmounts

**Default Proxmox unprivileged mapping:** container UID 0 → host UID 100000, container UID 65535 → host 165535. Linear block of 64K.

**Jellyfin deb user (typical):** UID 109 inside LXC → host UID 100109.

**Host `/mnt/pve/wdc` + `/mnt/pve/sea` ownership:** unknown from repo — must probe:
```bash
ssh root@tower 'stat -c "%U:%G %u:%g %a" /mnt/pve/wdc /mnt/pve/sea'
# And sample file inside:
ssh root@tower 'stat -c "%U:%G %u:%g %a" /mnt/pve/wdc/<sample-file>'
```

**Current LXC 100 (privileged? check):** LXC 100's conf shows NO `unprivileged: 1` line → **LXC 100 is privileged**. That's why PUID=1000/PGID=1000 in docker-compose works: container UID 1000 = host UID 1000 directly (no shift). Media files likely owned by UID 1000 on host.

**With unprivileged LXC 101 (UID shift +100000):**
- Jellyfin (uid 100109 on host) trying to read a file owned by uid 1000 on host → **READ FAILS** unless file is world-readable.
- Default mode on media files from *arr downloads: typically `0644` (world-readable) → **reads likely work**.
- Directory traversal requires `+x` on every parent → typically `0755` → also fine.

**Three paths forward (pick one during planning):**
1. **Rely on world-readable perms** (RO bindmount + 0644 files). Zero config. **Verify post-migration by `sudo -u jellyfin ls /media/wdc/movies | head`.** Lowest-effort path.
2. **`lxc.idmap` shift for specific GID:** map container gid 1000 → host gid 1000. Requires adding to `/etc/subuid` + `/etc/subgid` on tower, plus stanzas in `101.conf`:
   ```
   lxc.idmap: u 0 100000 1000
   lxc.idmap: u 1000 1000 1
   lxc.idmap: u 1001 101001 64535
   lxc.idmap: g 0 100000 1000
   lxc.idmap: g 1000 1000 1
   lxc.idmap: g 1001 101001 64535
   ```
   And add `jellyfin` user to gid 1000 (typically `users` group).
3. **Chmod host files to world-readable** (no LXC config change). Safest if files are already mostly 0644 — just fix stragglers.

**Recommendation:** Try Path 1 first (no config change). If it works, done. If it fails, try Path 3 before Path 2. Path 2 is the last resort because idmap changes require container restart and edit storm in `/etc/pve/lxc/101.conf`.

## Pitfalls

### Pitfall 1: SQLite corruption during rsync
**What goes wrong:** Copying `library.db` while Jellyfin has it open (WAL + SHM files mid-write) produces an inconsistent copy. Target Jellyfin then fails to start or silently drops data.
**Why it happens:** SQLite WAL mode keeps the DB file stable but leaves writes in the WAL; if rsync captures DB without WAL (or vice versa), state is torn.
**How to avoid:** Stop container with `docker stop` (SIGTERM, graceful), verify process exited cleanly, only then rsync. Never use `docker kill`, never rsync a running Jellyfin's data dir.
**Warning signs:** Jellyfin on LXC 101 logs `SQLite error: database disk image is malformed` on startup. → halt, restore from backup, retry.

### Pitfall 2: Permissions drift after chown
**What goes wrong:** `chown -R jellyfin:jellyfin /var/lib/jellyfin` but modes stay at 0644/0755 from rsync — fine. But plugin bin dirs may need 0700 for API keys.
**Why it happens:** Plugin configs sometimes contain secrets expecting restricted perms.
**How to avoid:** After chown, `find /var/lib/jellyfin -name '*.json' -path '*plugin*' -exec chmod 0600 {} \;` as a defensive sweep.
**Warning signs:** Plugin start failures in journalctl ("cannot open key file").

### Pitfall 3: Jellyfin version mismatch (Docker image vs deb repo)
**What goes wrong:** Docker `jellyfin/jellyfin:latest` on 2026-04-14 pinned to some sha; deb repo may be newer. If major.minor differs, DB schema migrates forward — rollback to Docker then requires restoring DB backup.
**Why it happens:** Jellyfin releases monthly; 10.10.x → 10.11.x schema evolves.
**How to avoid:** Pre-migration: `docker inspect jellyfin | jq '.[0].Config.Image'` and `docker exec jellyfin jellyfin --version`. Pin deb install: `apt install jellyfin=<exact>-1`. Keep DB backup regardless.
**Warning signs:** First-start log shows "Running migration 00xx of yy"; after this, rollback requires backup restore.

### Pitfall 4: tmpfs too small for 4K HEVC
**What goes wrong:** 4 GB tmpfs fills during a 4K HEVC transcode session (if Jellyfin buffers many segments). Jellyfin aborts playback with "out of disk space".
**Why it happens:** Jellyfin HLS segmenting writes segments forward of play head; can retain 30–60s of segments = several GB for 4K.
**How to avoid:** Monitor during D-17 4K test: `watch -n 2 df -h /var/cache/jellyfin/transcodes`. If approaches 80%, bump to 8G (keeping 8G RAM ceiling in mind).
**Warning signs:** "No space left on device" in jellyfin-ffmpeg stderr.

### Pitfall 5: Router forward points at wrong tower IP after reconfiguration
**What goes wrong:** Router forwards :22098 to `tower`'s LAN IP. If tower's LAN IP changes (DHCP lease churn, switch reboot), the forward breaks.
**Why it happens:** LAN IPs not statically assigned via DHCP reservation.
**How to avoid:** Confirm tower LAN IP is a DHCP reservation on the router, or use a static-IP config. Not in scope of this phase but note in README.
**Warning signs:** External access fails but LAN access works.

### Pitfall 6: `dev0:` with wrong gid silently breaks transcode
**What goes wrong:** If `dev0: /dev/dri/renderD128,gid=104` but host render GID is actually 107, the device node inside LXC has gid 104 (doesn't exist or wrong group). Jellyfin user can't access → ffmpeg falls back to software transcode silently, user sees "buffering".
**Why it happens:** Render GID varies across Debian installs (107, 104, 993).
**How to avoid:** Probe `ssh root@tower 'getent group render'` at plan time. Put the value into an Ansible variable used in `pct set 101 --dev0`. Add a smoke task to verify `stat -c %g /dev/dri/renderD128` inside LXC matches `getent group render` inside LXC.
**Warning signs:** HW accel not engaged in Jellyfin dashboard; `intel_gpu_top` on tower idle during playback.

### Pitfall 7: LinuxServer path assumption
**What goes wrong:** Research guides online assume `/config/data/data/library.db` (LinuxServer layout). But this repo runs `jellyfin/jellyfin:latest` (OFFICIAL) where DBs are at `/config/data/library.db`.
**Why it happens:** Docker image naming collides; LinuxServer is Google-famous.
**How to avoid:** Before rsync, `ssh root@docker-tower 'ls -la /opt/docker-configs/shared/jellyfin/'` and verify the top-level contents match the official layout in §Key Findings #4. Adjust rsync source paths accordingly.
**Warning signs:** rsync target `/var/lib/jellyfin/data/` ends up empty or double-nested.

## Security Domain

### Applicable ASVS Categories

| ASVS Category | Applies | Standard Control |
|---------------|---------|-----------------|
| V2 Authentication | yes | Jellyfin's built-in user auth; no changes this phase |
| V3 Session Management | yes | Jellyfin cookies + WebSocket session; standard |
| V4 Access Control | yes | Unprivileged LXC isolates service; RO bindmounts enforce storage separation; `dev0:` uid/gid confines GPU access to jellyfin user |
| V5 Input Validation | yes | Jellyfin handles; proxy adds `X-Forwarded-*` headers (reverse proxy config) |
| V6 Cryptography | yes | TLS termination (see Open Q 2); use Let's Encrypt or existing cert — never hand-roll |
| V10 Config | yes | `apt-mark hold` + Ansible-managed updates; SOPS for any secrets |
| V12 Files + Resources | yes | RO bindmounts prevent Jellyfin from writing to source media; tmpfs is non-persistent + mode 0755 |

### Known Threat Patterns

| Pattern | STRIDE | Mitigation |
|---------|--------|-----------|
| Unpatched Jellyfin on public port | T / E | `apt-mark hold` + Ansible update cadence; monitor Jellyfin security advisories |
| WebSocket hijack via plain HTTP | T / I | TLS (Open Q 2) + WSS |
| Container escape via /dev/dri | E | Unprivileged LXC + `dev0:` device node, not full `/dev/dri` bind (less surface than legacy approach) |
| Media filesystem write from compromised Jellyfin | T | `ro=1` on mp0/mp1 — enforced at kernel level |
| SQLite data exposure on backup | I | Ensure backup tarball `chmod 0600` if it leaves the LXC |
