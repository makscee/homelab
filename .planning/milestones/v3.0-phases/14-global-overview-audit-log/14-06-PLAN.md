<objective>
Fix the bun:sqlite shim so it proxies to native bun:sqlite when running under Bun, while
preserving the build-time throw stub for Next.js's Node.js worker. This unblocks /audit
(Gap 1, test 3, BLOCKER) and — as a downstream consequence — the Claude usage cards on
/overview (Gap 3, test 6, MAJOR).

Root cause: next.config.mjs webpack externals callback returns `commonjs ${absShimPath}`,
which hardcodes the absolute shim file as the require() target in the compiled bundle.
Under Bun runtime the bundle therefore loads apps/admin/lib/bun-sqlite-shim.js (not native
`bun:sqlite`), and every `new Database(...)` throws "bun:sqlite shim: should not be called
at build time".

Fix shape (simplest surgical change per gap context — no new abstractions, no refactor of
the webpack externals path): make the shim itself conditionally re-export native
`bun:sqlite` when the `Bun` global is present. The Node build worker has no `Bun` global,
so it falls through to the existing throw stub and the build continues to succeed.

Purpose: unblock UAT tests 3, 4, 6. Restore /audit and Claude usage cards.
Output: updated shim (.js + .ts), deployed build, human-verified /audit + /overview.
</objective>

Replace the body of `apps/admin/lib/bun-sqlite-shim.js` with the proxy pattern:

```js
/**
 * bun:sqlite shim — dual-mode.
 *
 * 1. Runtime (Bun): proxies native `bun:sqlite` by re-exporting its module.
 *    The webpack externals interceptor in next.config.mjs rewrites
 *    `import { Database } from 'bun:sqlite'` to `require('/abs/.../bun-sqlite-shim.js')`;
 *    under Bun, this file in turn `require('bun:sqlite')` which resolves natively.
 *
 * 2. Build (Node.js — Next.js worker): `Bun` global is absent; we export a
 *    throw-on-construct stub so `next build` can resolve the module without
 *    actually executing any SQLite code.
 *
 * Do NOT move the `typeof Bun` check below the native require — it must gate it.
 */
if (typeof Bun !== "undefined") {
  // Under Bun: delegate every export to the real native module.
  module.exports = require("bun:sqlite");
} else {
  // Under Node.js build worker: construction throws; bare require() does not.
  class Database {
    constructor(_path, _opts) {
      throw new Error("bun:sqlite shim: should not be called at build time");
    }
  }
  module.exports = { Database };
}
```

Align `apps/admin/lib/bun-sqlite-shim.ts` so the type surface matches what the runtime exports (no behavior change needed at build — webpack externals routes to the .js anyway — but keep TS in sync for tsserver / unit tests):

```ts
/**
 * bun:sqlite TS type surface. Runtime behavior lives in bun-sqlite-shim.js
 * (webpack externals routes the actual require() to the .js module).
 * This file exists for tsserver and direct-import test scenarios.
 */
export class Database {
  constructor(_path: string, _opts?: unknown) {
    if (typeof (globalThis as any).Bun !== "undefined") {
      // If someone imports the .ts at runtime under Bun, defer to native.
      // eslint-disable-next-line @typescript-eslint/no-require-imports
      const native = require("bun:sqlite") as { Database: new (p: string, o?: unknown) => unknown };
      return new native.Database(_path, _opts) as unknown as Database;
    }
    throw new Error("bun:sqlite shim: should not be called at build time");
  }
  run(_sql: string, ..._params: unknown[]): unknown { return null; }
  prepare(_sql: string): unknown { return null; }
  query(_sql: string): unknown { return null; }
  close(): void {}
}
```

Do NOT edit `next.config.mjs`. The externals interceptor stays as-is — it still routes `bun:sqlite` imports through the shim file; the fix is that the shim now does the right thing at runtime.

Do NOT change audit-db.server.ts or token-registry.server.ts — same import contract.
