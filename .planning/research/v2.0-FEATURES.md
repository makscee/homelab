# Features — Claude Code Usage Monitor (v2.0)

**Milestone:** v2.0 — Claude Code Usage Monitor
**Researched:** 2026-04-16
**Overall confidence:** MEDIUM-HIGH — endpoint schema corroborated across 3 independent community sources; Max 20x mechanics confirmed via Anthropic Help Center + multiple GitHub issues. The OAuth usage endpoint itself is **undocumented** and has a known 429-storm bug — this is the dominant feasibility risk.

---

## Max 20x Subscription Mechanics

Cited specifics driving what the exporter can actually measure.

### Subscription tier
- **Max 20x** = $200/mo, 20× Pro base usage band. Max 5x ($100) is the cheaper tier referenced in PROJECT.md — **confirm actual subscription on each token** before assuming 20× capacity. [Anthropic Help Center — Max plan](https://support.claude.com/en/articles/11049741-what-is-the-max-plan)
- Approximate capacity: **~220k tokens per 5-hour window**, 50–800 prompts/window depending on prompt size. [Northflank](https://northflank.com/blog/claude-rate-limits-claude-code-pricing-cost), [NxCode](https://www.nxcode.io/resources/news/claude-code-pricing-2026-free-api-costs-max-plan)

### Session window (5-hour) — rolling, idle-paused
- **Rolling**, not calendar-fixed. Clock starts on first `claude` invocation / first message; resets only on the first message sent **after** the 5h elapses. Idle does not tick the clock. [SitePoint](https://www.sitepoint.com/claude-code-rate-limits-explained/)
- Returned as `five_hour.utilization` (0.0–1.0) + `five_hour.resets_at` (ISO 8601).

### Weekly window (7-day) — fixed per-account anchor
- Resets on a fixed 7-day cycle from per-account cycle start (not a global calendar week). Community reports a common **Thursday 8pm PT** anchor. [Truefoundry](https://www.truefoundry.com/blog/claude-code-limits-explained)
- Returned as `seven_day.utilization` + `seven_day.resets_at`.

### Model split (post-2025-11-24 change)
- Since Nov 24 2025, **Sonnet has its own separate weekly cap** in addition to the combined all-models cap — surfaced as `seven_day_sonnet.utilization` in the usage payload.
- **Opus is NOT separately broken out** in the response — it consumes from the combined `seven_day` bucket at a higher weight (~5× Sonnet historically). Heavy Opus burns `seven_day` while `seven_day_sonnet` stays low — inferrable but not authoritative. [GitHub #12487](https://github.com/anthropics/claude-code/issues/12487), [Medium / Gunratna](https://medium.com/@Gunratna/what-does-more-usage-in-claude-4-5-limits-actually-mean-explained-for-opus-sonnet-haiku-720535b70d55)
- **Haiku is not separately tracked** in the OAuth payload.

### What counts toward quota
- **Input + output tokens weighted by model.** Cache reads/writes counted at reduced weight. Not raw message count. Each tool-use round trip and sub-agent invocation counts.

### Behavior at 100%
- **Hard cutoff.** Session 100% → requests return `rate_limit_error` until `resets_at`. Weekly 100% → locked until weekly reset, no degradation to cheaper model, no soft mode. [Portkey](https://portkey.ai/blog/claude-code-limits/)

### The critical feasibility caveat — endpoint behavior
- **Endpoint:** `GET https://api.anthropic.com/api/oauth/usage`
- **Required headers:** `Authorization: Bearer <oauth-token>` + `anthropic-beta: oauth-2025-04-20` (missing → 401)
- **Token source:** `~/.claude/.credentials.json` on the host running `claude` (refreshed automatically by the CLI)
- **Undocumented** — subject to silent change. Treat as best-effort signal, not contract.
- **Known 429-storm bug:** endpoint aggressively rate-limits polling. Reports of persistent 429 with `retry-after: 0` indefinitely, even at 30–300s intervals. [GitHub #31021](https://github.com/anthropics/claude-code/issues/31021), [GitHub #31637](https://github.com/anthropics/claude-code/issues/31637), [GitHub #30930](https://github.com/anthropics/claude-code/issues/30930)
- **Operational implication:** scrape interval must be ≥120s, ideally 180–300s, with jittered backoff and sticky-last-known-value on 429. Do NOT inherit Prometheus `scrape_interval: 15s`.
- **Example response schema (observed in community tools):**
  ```json
  {
    "five_hour":        { "utilization": 0.42, "resets_at": "2026-04-16T17:00:00Z" },
    "seven_day":        { "utilization": 0.61, "resets_at": "2026-04-18T04:00:00Z" },
    "seven_day_sonnet": { "utilization": 0.35, "resets_at": "2026-04-18T04:00:00Z" }
  }
  ```

## Differentiators (nice to have)

Deferrable without blocking v2.0 close. Ordered by value-to-effort.

| # | Feature | Complexity | Value |
|---|---------|------------|-------|
| D1 | **Sonnet-specific weekly gauge** (`seven_day_sonnet`) as separate panel | SMALL | High — already in payload, essentially free; shows Opus-vs-Sonnet burn split |
| D2 | **Burn-rate prediction** — "at current rate you'll hit weekly cap at HH:MM" via `predict_linear(claude_code_weekly_used_ratio[6h], resets_seconds)` | SMALL-MEDIUM | High — turns the dashboard from observational to actionable |
| D3 | **Fleet roll-up** — `avg(claude_code_weekly_used_ratio)` as "fleet average weekly burn" | SMALL | Medium — marginal at 2–5 tokens |
| D4 | **Cost estimation** — `claude_code_estimated_cost_usd = utilization × Max_20x_implied_capacity_usd` | MEDIUM | Medium — fragile; Anthropic doesn't publish Max-tier $/token; best-effort with caveat |
| D5 | **Anomaly alert** — `rate(claude_code_weekly_used_ratio[30m]) > historical_avg × 3` | MEDIUM | Medium — catches runaway agent loops |
| D6 | **Per-token comparison chart** — stacked timeseries "which worker burns most" | SMALL | Medium — identifies greedy LXC |
| D7 | **Opus-share inference** — derived `(seven_day − seven_day_sonnet × weight)` as Opus proxy | MEDIUM | Low-Medium — unofficial inference; flag uncertainty in panel description |
| D8 | **Session annotations** — Grafana timeline markers "new 5h session started" | MEDIUM | Low — nice, limited operational value |
| D9 | **429-rate counter** — `claude_code_oauth_429_total` for endpoint-health observability | SMALL | Medium — critical for diagnosing false-zero readings; promote to table-stakes if Phase 1 shows severe 429 storms |
| D10 | **Status-line HTTP endpoint** (à la ohugonnot/claude-code-statusline) for worker terminals | MEDIUM | Low — out of scope of operator dashboard, but useful for LXC users |

## Anti-features (explicitly OUT)

| Anti-feature | Why avoided | What to do instead |
|--------------|-------------|-------------------|
| **Conversation content / prompt logging** | Privacy violation; operator cares how-much, not what-was-said | OAuth endpoint only — never touch conversation transcripts |
| **Automated billing / chargeback** | No multi-tenant billing at 6-node scale; adds financial-systems surface | Manual review of cost panel (D4) if it ships |
| **Cross-token load redistribution** (auto-failover) | Explicitly deferred to Shape B per PROJECT.md line 39 | Operator sees burn, manually hands off |
| **Writes/mutations against Anthropic APIs** | Read-only observability; `oauth/usage` is GET-only anyway | N/A |
| **Token rotation automation** | PROJECT.md Shape A — manual rotation (edit SOPS + redeploy) | Document runbook |
| **OpenTelemetry Collector pipeline** (à la ColeMurray/claude-code-otel) | Adds OTel Collector + dual-write + worker env-var instrumentation; homelab already on direct Prometheus scrape | Plain Python exporter using `prometheus_client` |
| **Multi-user RBAC on dashboard** | 2–5 tokens, one operator — no tenancy need | Token-selector variable (T8) suffices |
| **Non-Telegram alert channels** (Slack/Discord/PagerDuty) | v1.0 converged on Telegram; no reason to fork | Reuse Alertmanager route |
| **Haiku / generic Anthropic API keys** | Out of scope — this milestone is Claude Code OAuth quotas only | Separate project if ever needed |

**Decisive insight:** every Anthropic-official and community "Claude Code observability" project uses Claude Code's built-in **OpenTelemetry output** (session/event level), NOT the `/api/oauth/usage` quota endpoint. This milestone's differentiator is **fleet-wide quota monitoring of 2–5 tokens from a central exporter without touching the workers** — a real gap in the ecosystem.

## Feature Dependencies

```
T1 (SOPS registry)
 └─► T2 (exporter) — loads tokens from SOPS at startup
     ├─► T3 (reset countdown) — same scrape
     ├─► T4, T5, T6, T7 (dashboard panels) — need metrics flowing
     │    └─► T8 (token variable) — needs labelled metrics
     ├─► T9, T10, T11, T12, T13 (alert rules) — need metrics to evaluate
     │    └─► T14 (Telegram routing) — reuses v1.0 path
     └─► T16 (Prometheus scrape) — configures the pull

T2 → T15 (Ansible) packages T2 + T3 for reproducible deploy.

Differentiators all depend on T2 being stable:
  D1 (Sonnet gauge)        — needs T2 exposing seven_day_sonnet
  D2 (burn-rate predict)   — needs T7 (historical data accumulated)
  D4 (cost)                — needs T2 + per-tier pricing config
  D9 (429 counter)         — part of T2's error handling; promote to table-stakes if Phase 1 proves 429 storm severe
```

**Critical path:** T1 → T2 → (T4–T8 ‖ T9–T14) → T15 → T16.

**Phase 1 research gate:** before T2, prove (a) OAuth endpoint reachable from mcow over Tailnet, (b) jittered ≥180s polling across 2–5 tokens avoids the 429 storm. If not, pivot to "local token counter" fallback noted in PROJECT.md (exporter reads token-usage JSONL written by each worker's Claude CLI, sidestepping Anthropic entirely).
